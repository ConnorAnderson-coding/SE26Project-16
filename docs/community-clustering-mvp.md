# 社区聚类迭代 2 MVP 设计

## 1. 文档目的

本文记录社区聚类迭代 2 的产品与技术边界，并同步阶段 3D 只读 HTTP API 的当前实现状态。规划内容不等同于已经可调用的接口；公开 API 的准确字段和错误契约以 `community-clustering-api.md` 为准。

本文中的“运行（run）”指一次使用固定输入快照和参数完成的聚类任务；“版本（version）”指该次运行生成且全局唯一的结果版本；“社区”是 K-Means 产生的一个硬聚类簇。

## 2. 当前仓库实现现状

根据阶段 3D 当前源码，仓库情况如下：

- 前端 `campus-activity` 已有“社区聚类”路由和展示页面，但页面读取 `mockData.js` 中的静态社区、成员和坐标，仅用于演示，尚未接入真实聚类 API。
- Spring Boot 后端位于 `campus-activity/backend`，当前使用 H2；聚类运行、社区、社区成员的持久化与同步编排基础已经存在。
- 用户数据已包含学院、年级、兴趣标签和可参与时间；活动数据已包含活动类别；报名、收藏、签到和评价记录可支持首版行为特征聚合。
- 当前没有浏览日志或浏览行为实体，因此首版特征不得包含访问次数、停留时长、点击轨迹等浏览数据。
- 已实现三个只读公开 API：管理员运行详情、最新成功版本概览、当前用户归属。
- 当前认证边界已经使用 Spring Security HTTP Session、`JSESSIONID` 与 CSRF；管理员查询要求 `ROLE_ADMIN`，用户查询要求已认证。
- `POST /api/v1/admin/community-clustering/runs`、管理员成员分页、真正的异步任务提交与恢复尚未实现。
- 当前 `CommunityClusteringOrchestrator` 是同步阻塞执行，不能直接作为返回 `202 Accepted` 的 Controller。

## 3. 迭代 2 社区聚类 MVP 目标

迭代 2 的目标是基于仓库真实可获得的用户画像与活动行为，以确定性 K-Means 硬聚类生成版本化社区，并提供可由现有前端后续接入的社区概览、二维散点坐标和个人归属查询能力。

MVP 必须满足：

- 每次运行采用 K-Means 硬聚类，同一运行版本中一个用户恰好属于一个社区。
- `2 <= K <= 有效用户数`；本地演示在未显式传入 K 时默认 `K=2`。
- K-Means 固定使用 `random_state=42`，以支持可重复测试。
- 使用 PCA 将标准化后的特征降到二维；每次运行内分别将 x、y 坐标归一化到 `[0, 100]`。投影维度精确无变化，或按 API 契约的绝对容差 `1e-12`、相对容差 `1e-9` 判定为数值上近零范围时，该维统一为 50；零解释方差比例返回 0，不得产生 `NaN` 或无穷大。
- 浏览器只调用 Spring Boot 公开 API，绝不直接调用 Python 服务。
- Spring Boot 负责最终权限校验、运行编排、数据聚合、输入快照组织、结果校验、社区 ID 与展示元数据生成和事务持久化。
- Python 负责特征预处理、特征编码、标准化、K-Means 和 PCA，不负责公开权限或业务数据持久化。
- 最近成功版本在新运行失败时保持可用。

## 4. 纳入范围

### 4.1 首版特征来源

首版只能使用仓库目前能够采集或聚合的数据：

| 特征 | 来源与建议口径 |
| --- | --- |
| 兴趣标签 | 用户 `interests`，多值类别特征 |
| 学院 | 用户 `college`，单值类别特征 |
| 年级 | 用户 `grade`，单值类别特征 |
| 可参与时间 | 用户 `availableTime`，多值类别特征 |
| 报名次数 | 用户全部报名记录数 |
| 审核通过次数 | 报名状态为审核通过的记录数；实现阶段需与现有状态字面量统一 |
| 收藏次数 | 用户收藏记录数 |
| 签到次数 | 用户签到记录数 |
| 评价次数 | 用户评价记录数 |
| 平均评分 | 用户已有评分的算术平均值；无评价时按预处理规则填充，并保留一致的数据口径 |
| 各活动类别参与次数 | 按活动类别统计审核通过的报名次数；类别集合由本次输入快照确定 |

“有效用户”在 MVP 中指：存在有效用户账号、不是管理员角色、能够完成上述特征映射且未因数据完整性错误被剔除的用户。学生和组织者均可纳入。兴趣、学院、年级、可参与时间或行为记录缺失时，应按固定预处理规则编码为空类别或零值，不能随意虚构数据；仅当记录无法关联到有效用户或出现不可恢复的数据错误时才剔除，并记录样本数量和原因。

### 4.2 功能范围

以下列表是迭代 2 的总体范围，不表示各项都已实现。阶段 3D 已实现三个只读查询；管理员触发、成员分页和异步恢复仍是规划内容。

- 管理员手动提交 K 值并触发一次聚类运行（规划中，尚未实现公开 POST）。
- 管理员查询运行状态和结构化失败摘要（已实现）。
- Spring Boot 聚合真实业务数据并调用内部 Python 接口。
- Python 完成确定性预处理、标准化、K-Means、PCA 和坐标归一化。
- Spring Boot 校验结果并以版本化方式保存运行、社区和成员。
- 登录用户查询最新成功版本的社区概览、匿名散点坐标和自己的社区归属（已实现）。
- 管理员分页查询指定社区的成员（规划中，尚未实现；个人资料范围待隐私评审）。
- 新运行失败时继续提供最近一次成功版本。
- 为后续 H2 到 MySQL 迁移保留明确的数据类型与约束设计。

## 5. 明确排除范围

第一版明确不实现：

- Redis、分布式缓存或分布式锁。
- 社区留言板、聊天或群组社交功能。
- t-SNE 或其他替代降维算法。
- 定时任务、自动重跑或周期性聚类。
- 聚类版本变化后的迁移通知。
- 社区推荐、活动推荐或好友推荐。
- MySQL 迁移；本地仍以 H2 为实现基线。
- 浏览行为特征；当前仓库没有浏览日志，不得制造或推断浏览数据。
- 浏览器直连 Python 服务。
- 在阶段 0 新建或改造登录、令牌或会话体系。
- 软聚类、多社区归属、人工调整成员归属。

## 6. 角色与权限语义

当前只读 API 与后续规划 API 统一遵循以下权限语义：

| 角色/主体 | 能力 | 限制 |
| --- | --- | --- |
| 管理员 | 当前可查询任意运行状态；手动触发聚类和成员分页仍在规划中；也可使用普通用户查询能力 | 管理员端点要求 `ROLE_ADMIN`；不能信任 Body、Query 或 Header 中的 `role`、`createdBy` 或用户标识 |
| 已登录普通用户 | 查询最新成功版本的社区概览和匿名二维坐标；查询自己在最新成功版本中的社区 | 端点要求 `authenticated`；`latest` 和 `me` 的当前用户只来自 `Authentication.getName()`，不接受客户端传入 `userId` |
| Spring Boot | 访问内部 Python 接口，发送最小必要特征快照，校验并持久化结果 | Python 地址和内部调用凭据只存在于服务端配置，不得返回浏览器 |
| Python 聚类服务 | 接收 Spring Boot 的内部请求并返回计算结果、提供内部健康检查 | 不面向浏览器，不负责用户权限、公开 API 或数据库持久化 |

浏览器使用 HTTP Session，Cookie 为 `JSESSIONID`，跨源请求需要发送凭据。写请求需要 CSRF Token，GET 查询不要求 CSRF Token。认证与授权错误码统一为 `AUTHENTICATION_REQUIRED`、`ACCESS_DENIED`、`CSRF_TOKEN_INVALID` 和 `INVALID_CREDENTIALS`；错误响应的 `details` 当前固定为 `{}`。

## 7. 目标业务流程（含尚未实现的异步提交）

以下流程描述完整目标。步骤 1 至 4 所需的公开 POST、后台执行器、提交拒绝处理和恢复机制尚未实现；当前 Orchestrator 会同步阻塞执行，不能把执行完成后的响应描述为 `202 Accepted`。

1. 管理员通过 Spring Boot 的公开 API 手动提交 `clusterCount`；未提供时，本地演示默认使用 2。
2. Spring Boot 使用可信服务端身份校验管理员权限，统计有效用户数，并校验 `2 <= K <= 有效用户数`。
3. 目标实现中，Spring Boot 创建全局唯一版本的 `ClusteringRun`，状态为 `PENDING`，保存算法、K、`randomState=42`、特征模式版本和参数快照；登记事务提交后才返回 `202 Accepted`。此行为当前尚未实现。
4. 执行器领取任务后，在短事务中将状态从 `PENDING` 更新为 `RUNNING`，记录 `startedAt`。
5. Spring Boot 从用户、活动、报名、收藏、签到和评价数据聚合本次输入快照。整个快照使用同一数据口径，记录最终 `sampleCount`，不包含密码、认证令牌或其他秘密。
6. Spring Boot 仅从服务端调用 `POST /internal/v1/clustering/run`，传递运行标识、固定参数、特征模式版本及用户特征行。浏览器不接触内部地址。
7. Python 按 `featureSchemaVersion` 执行缺失值处理、类别编码和数值特征处理，再进行标准化。
8. Python 使用 `random_state=42` 执行 K-Means 硬聚类，保证每个输入用户获得且只获得一个 `clusterNo`，并计算成员到聚类中心的距离。
9. Python 对同一标准化特征矩阵执行 PCA 得到二维坐标，分别将 x、y 线性归一化至 `[0, 100]`。若某一投影维度所有值相同，或按 API 契约的固定绝对/相对容差判定为数值上退化，该维统一返回 50；零方差主成分的解释方差比例返回 0。若不同特征行少于 K，或应用退化规则后仍产生非有限结果，返回契约定义的 `CLUSTERING_COMPUTATION_FAILED`。
10. Python 返回社区编号、确定排序且最多 3 项的代表性兴趣、成员归属、坐标、中心距离和非敏感指标；不生成社区 ID、名称、描述或颜色，也不直接写业务数据库。
11. Spring Boot 校验返回结果：运行标识匹配、社区数为 K、用户集合与输入一致、每个用户恰好出现一次、簇编号合法、坐标范围合法、数值有限且无秘密字段。
12. 校验通过后，Spring Boot 按 API 契约为每个簇生成不透明 `communityId`、`社区 {clusterNo + 1}` 名称、仅基于 `topInterests` 的描述和固定调色板颜色；随后在一个数据库事务中保存全部 `Community` 和 `CommunityMember`，同步社区成员数与摘要，并把运行更新为 `SUCCESS`、写入 `finishedAt` 和指标。提交后该版本才有资格成为 latest。
13. 若聚合、内部调用、计算、校验或持久化任一步失败，Spring Boot 回滚未完成的社区和成员写入，在独立短事务中把运行标记为 `FAILED` 并保存脱敏错误摘要。最近一次 `SUCCESS` 版本不受影响。
14. 已登录用户通过 Spring Boot 查询最新成功版本的社区概览与匿名散点，或查询自己的唯一社区；管理员可查询运行状态和社区成员。

## 8. 验收标准

以下验收标准包含后续目标；标为规划中的能力不能用当前只读 API 验收。

### 8.1 业务与算法

- 管理员可手动触发聚类，省略 K 时本地演示使用 2（规划中，尚未实现公开 POST）。
- K 小于 2 或大于有效用户数时，请求被拒绝且不启动 Python 计算。
- 同一输入、同一特征模式和 K 下，固定 `random_state=42` 的测试结果可重复。
- 每个成功版本的有效用户数等于成员记录数，且同一用户在同一运行中只有一个社区。
- 聚类算法为 K-Means 硬聚类，二维可视化算法为 PCA。
- 所有 x、y 都是有限数且分别位于 0 到 100（含边界）。
- PCA 单维精确退化或在固定容差内数值退化时，该维坐标全部为 50、对应零方差主成分的解释方差比例为 0；全退化或不同特征行不足以形成 K 个非空簇时，行为与错误码符合内部 API 契约。
- 特征只来自已列明的真实数据源，不包含浏览数据。

### 8.2 架构与安全

- 浏览器网络请求中不存在 Python 内部接口调用。
- 公开 API 只由 Spring Boot 提供；Python 服务不承担公开权限判断和持久化。
- 管理员接口拒绝非管理员，`latest` 与 `me` 只从 `Authentication.getName()` 确定当前用户。
- HTTP Session 使用 `JSESSIONID`；前端发送凭据，写请求携带 CSRF Token，GET 查询不要求 CSRF Token。
- 公开响应不包含密码、认证令牌、内部特征向量或 Python 服务地址。
- 内部调用失败时，错误信息经脱敏后记录，不暴露内部地址或凭据。

### 8.3 版本与一致性

- 运行状态仅使用 `PENDING`、`RUNNING`、`SUCCESS`、`FAILED`。
- 只有完整事务提交的 `SUCCESS` 运行可被 latest 查询选中。
- 新运行失败不会覆盖或隐藏最近一次成功结果。
- 社区、成员和运行成功状态要么一起提交，要么全部不生效。
- 管理员可以定位失败运行及其脱敏失败原因。

### 8.4 API 与展示

- 三个已实现公开 GET 和两个内部 Python API 的行为符合接口契约文档；规划 POST 与管理员成员分页不得计为已实现。
- 最新概览能返回社区摘要与 `[0, 100]` 二维坐标；普通用户看不到成员敏感详情。
- 前端可只使用一次 `latest` 响应渲染社区表格、图例和匿名散点：以 `communityId`、`pointId` 为键，使用 `memberCount`、`points[].x/y` 和 `currentUser`，不得再按匿名点关联用户列表。
- 当前用户不在最新样本中时，`me` 返回明确的空归属语义，而不是伪造社区。
- 没有成功运行时，latest 和 me 返回统一、可识别的错误。

## 9. 已知限制

- 当前样本规模、初始化演示数据和行为记录较少，聚类质量指标不等同于真实校园环境效果。
- K-Means 偏好近似球状、方差相近的簇，对类别特征编码和稀疏高维特征较敏感。
- PCA 只保留二维线性投影，散点距离不能完整表达高维相似度。
- 社区编号本身没有跨版本稳定语义；社区名称和颜色只能是该版本内的展示元数据，不能据此推断用户身份或长期群体迁移。
- 缺失画像和低行为量会降低区分度；填充值策略必须版本化，不能把缺失解释为真实偏好。
- 真正异步的 POST 仍缺少后台执行器、拒绝处理、重启恢复与失败可观测性；当前同步 Orchestrator 不能伪装成 `202` 接口。
- 管理员成员分页若公开 `userId`、姓名、学院或年级，仍需先完成隐私与最小权限决策。
- H2 与未来 MySQL 在 JSON、约束、索引、时间精度和标识符规则上存在差异，需要迁移验证。
- MVP 不提供自动调 K、社区质量人工审核或跨版本社区对齐。

## 10. 分阶段实施计划

| 阶段 | 交付物 | 边界与退出条件 |
| --- | --- | --- |
| 阶段 0：设计与契约 | MVP、API、数据模型三份文档 | 已完成初始设计；文档随后需持续同步实现 |
| 阶段 1：数据持久化骨架 | 三个后端实体、仓库、约束、状态枚举与持久化测试 | 不接 Python，不改前端；验证版本、成员唯一性及事务规则 |
| 阶段 2：Python 计算核心 | FastAPI 内部接口、特征预处理、标准化、K-Means、PCA、归一化及 pytest | 固定 `random_state=42`；以契约样本验证一人一社区和坐标范围 |
| 阶段 3：Spring 编排与公开 API | 特征聚合、内部客户端、任务状态、结果校验、事务落库与公开 API | 阶段 3D 已交付三个只读 GET；POST、管理员成员分页和真正异步恢复仍待后续独立阶段 |
| 阶段 4：前端真实接入 | 替换社区页 mock 数据，展示运行状态、社区概览、散点和个人归属 | 将 mock 的 `id/members/userId` 读取改为 `communityId/points/pointId`；匿名 tooltip 不查询成员资料；提供空态、加载态和错误态，不暴露内部字段 |
| 阶段 5：联调与验收 | H2 本地端到端联调、接口契约测试、前后端与 Python 全套检查 | 完成本仓库要求的测试、lint 和 build；记录剩余风险 |
| 后续阶段 | MySQL 迁移、可观测性及明确获批的新能力 | 不属于第一版，需单独设计和评审 |
