# 智能分析与建议模块 — 软件详细设计文档

> 版本：v1.0 | 日期：2026-07-13 | 项目：校园活动一站式服务平台

---

## 1. 概述

### 1.1 模块定位

"智能分析与建议模块"对活动的报名转化率、到场率、用户评价（评分+文字反馈）进行智能分析，自动生成下一次活动的改进建议供组织者参考。

### 1.2 设计原则

| 原则 | 说明 |
|------|------|
| **不破坏已有模块** | 所有新增文件均在独立包/路径下，不改动已有 Entity/Service/Controller 核心逻辑 |
| **离线优先** | LLM 调用通过定时任务异步触发，不阻塞用户请求 |
| **优雅降级** | LLM 调用失败时自动降级为规则模板生成，保证可用性 |
| **缓存加速** | 分析结果使用 Redis 缓存，TTL 可配 |
| **隐私保护** | 用户文字反馈传入 LLM 前做脱敏处理 |
| **可扩展** | 分析维度通过接口抽象，支持后续扩展传播路径等 |

### 1.3 技术选型

| 层面 | 技术 | 用途 |
|------|------|------|
| 统计计算 | Spring Boot + MySQL + JPA Native Query | 聚合统计查询 |
| 缓存 | Redis + Spring Cache | 缓存分析结果，减轻 DB 压力 |
| LLM | DeepSeek API (HTTP) | 生成改进建议 |
| 定时任务 | Spring `@Scheduled` | 离线触发分析 |
| 序列化 | Jackson JSON | LLM 请求/响应序列化 |
| HTTP 客户端 | Spring `RestClient` (Java 25 内置) | 调用 LLM API |

## 4. Service 层设计

### 4.1 `AnalyticsEngine` — 指标计算核心

```
┌─────────────────────────────────────────────────────────────────┐
│                       AnalyticsEngine                            │
│                                                                  │
│  computeMetrics(Long activityId) → ActivityMetrics               │
│  ┌──────────────────────────────────────────────────────────┐   │
│  │ 输入: 活动ID                                                │   │
│  │ 计算:                                                      │   │
│  │  ① signupRate = signupCount / maxParticipants  (来自 activity)│
│  │  ② attendanceRate = checkInCount / approvedCount           │   │
│  │       checkInCount  ← check_in 表 count by activityId      │   │
│  │       approvedCount ← registration 表 count status='approved'│  │
│  │  ③ avgRating = AVG(feedback.rating) WHERE activityId       │   │
│  │  ④ ratingDistribution = GROUP BY rating                    │   │
│  │  ⑤ checkInMethodsStats = GROUP BY method                   │   │
│  │  ⑥ favoriteConversion = signupCount / favoriteCount        │   │
│  │                                                             │   │
│  │ 辅助维度（扩展）:                                           │   │
│  │  ⑦ 同类活动历史平均指标对比                                 │   │
│  │  ⑧ 报名时间趋势（按小时统计）                               │   │
│  └──────────────────────────────────────────────────────────┘   │
│                                                                  │
│  对于辅助维度⑦⑧使用 Native Query 优化：                         │
│  @Query(value =                                                │
│    "SELECT AVG(CAST(a.signupCount AS DECIMAL)/a.maxParticipants)"│
│    + " FROM Activity a WHERE a.category = :category "          │
│    + " AND a.status = 'ended'", nativeQuery = true)            │
│  BigDecimal getCategoryAvgSignupRate(String category);         │
└─────────────────────────────────────────────────────────────────┘
```

### 4.2 `SummaryBuilder` — 数据摘要构建器

将 `ActivityMetrics` 格式化为结构化文本，作为 LLM 输入的上下文：

```text
## 活动信息
- 名称：{title}
- 类别：{category}
- 时间：{startTime} ~ {endTime}
- 地点：{location}
- 组织者：{organizerName} | 学院：{college}

## 核心指标
- 报名转化率：{signupRate}% — 报名 {signupCount}/{maxParticipants} 人
- 到场率：{attendanceRate}% — 签到 {checkInCount}/{approvedCount} 人
- 平均评分：{avgRating}/5.0 — 共 {feedbackCount} 条评价
- 评分分布：5★: {c5} | 4★: {c4} | 3★: {c3} | 2★: {c2} | 1★: {c1}
- 收藏转化率：{favoriteConversion}%

## 对比分析
- 同类活动({category})历史平均到场率：{categoryAvgAttendance}%
- 同类活动历史平均评分：{categoryAvgRating}/5.0

## 用户反馈摘要（已脱敏）
{top5_feedback_summaries}
```

**脱敏策略**：
- 去除姓名、学号等 PII 信息
- 评价原文超过 200 字符时截断
- 仅取评分与内容的组合，不关联用户 ID

### 4.3 `SuggestionGenerator` — 建议生成器

```
SuggestionGenerator
├── generateSuggestions(Activity activity, ActivityMetrics metrics)
│   │
│   ├─→ 步骤1: SummaryBuilder.build(activity, metrics)
│   │         → 构建结构化数据摘要
│   │
│   ├─→ 步骤2: 拼接 Prompt 模板 + 摘要
│   │         → 完整 LLM 请求体
│   │
│   ├─→ 步骤3: LlmClient.generateImprovements(request)
│   │         → 调用 LLM API（带超时+重试）
│   │
│   └─→ 步骤4: 解析 JSON 响应 → List<SuggestionItem>
│       │
│       ├─ 成功 → 标记 source="llm"
│       │
│       └─ 失败/超时/异常
│           └─→ fallbackSummarize() → 标记 source="rule"
│               根据预置规则模板生成：
│               - 报名率 < 50% → "建议优化海报设计，增加多渠道宣传"
│               - 到场率 < 70% → "建议提前推送签到提醒"  
│               - 评分 < 4.0  → "建议收集具体反馈并优化活动流程"
```

### 4.4 `LlmClient` — LLM API 调用客户端

```
LlmClient（使用 Spring RestClient）
│
├─ 配置注入（来自 AnalyticsConfig / application.properties）
│    app.analytics.llm.api-url=https://api.deepseek.com/v1/chat/completions
│    app.analytics.llm.api-key=${DEEPSEEK_API_KEY}
│    app.analytics.llm.model=deepseek-chat
│    app.analytics.llm.timeout=30s
│    app.analytics.llm.max-retries=3
│
├─ generateImprovements(String prompt) → LlmResponse
│    │
│    ├─ HTTP POST → LLM API (application/json)
│    ├─ 超时: 30s
│    ├─ 重试: 3 次，间隔 2s / 5s / 10s 退避
│    └─ 响应解析: JSON → LlmResponse.suggestions
│
└─ 降级: 任何异常都抛出 LlmCallException
         → 上层 SuggestionGenerator 捕获后走 fallback
```

### 4.6 `AnalyticsService` — 分析服务组合层

负责串联上述子服务（不暴露给 Controller，是内部组合层）：

```java
@Service
public class AnalyticsService {
    private final AnalyticsEngine engine;
    private final SuggestionGenerator suggestionGenerator;
    private final ActivityAnalysisRepository analysisRepository;

    @Transactional
    public ActivityAnalysis analyze(Long activityId) {
        Activity activity = activityService.getActivityEntity(activityId);
        ActivityMetrics metrics = engine.computeMetrics(activityId);
        List<SuggestionItem> suggestions =
            suggestionGenerator.generateSuggestions(activity, metrics);

        ActivityAnalysis analysis = new ActivityAnalysis();
        analysis.setActivity(activity);
        analysis.setSignupRate(metrics.signupRate());      // 等等
        analysis.setSuggestions(suggestions);
        analysis.setGeneratedAt(LocalDateTime.now());
        return analysisRepository.save(analysis);
    }
}
```

## 6. 定时任务设计

### 6.1 `AnalysisScheduler`

```java
@Component
@RequiredArgsConstructor
public class AnalysisScheduler {

    private final AnalyticsService analyticsService;
    private final ActivityRepository activityRepository;
    private final AnalysisScheduleLogRepository scheduleLogRepository;

    /**
     * 每日凌晨 2:00 扫描最近3天结束的活动，自动触发分析
     */
    @Scheduled(cron = "0 0 2 * * ?")
    public void analyzeEndedActivities() {
        AnalysisScheduleLog log = new AnalysisScheduleLog();
        log.setJobName("analyzeEndedActivities");
        log.setStatus("STARTED");
        log.setStartedAt(LocalDateTime.now());
        scheduleLogRepository.save(log);

        try {
            LocalDateTime threeDaysAgo = LocalDateTime.now().minusDays(3);
            List<Activity> endedActivities = activityRepository
                .findEndedAfter(threeDaysAgo, PageRequest.of(0, 100));

            int successCount = 0, skipCount = 0;
            for (Activity act : endedActivities) {
                if (analysisRepository.existsByActivityId(act.getId())) {
                    skipCount++;
                    continue;
                }
                try {
                    analyticsService.analyze(act.getId());
                    successCount++;
                } catch (Exception e) {
                    // 单活动失败不中断整体流程
                    log.warn("活动分析失败: activityId={}, error={}",
                        act.getId(), e.getMessage());
                }
            }

            log.setStatus("SUCCESS");
            log.setFinishedAt(LocalDateTime.now());
            log.setDurationMs((int) Duration.between(log.getStartedAt(),
                log.getFinishedAt()).toMillis());
        } catch (Exception e) {
            log.setStatus("FAILED");
            log.setErrorMessage(e.getMessage());
            log.setFinishedAt(LocalDateTime.now());
        }
        scheduleLogRepository.save(log);
    }
}
```

### 6.2 调度策略

| 策略 | 说明 |
|------|------|
| **触发时机** | 每日凌晨 2:00 (Cron: `0 0 2 * * ?`) |
| **扫描范围** | 最近 3 天内结束、且尚未分析的活动 |
| **批量处理** | 每次最多 100 个活动，逐个分析 |
| **幂等保证** | 检查 `activity_analysis` 唯一索引，已存在则跳过 |
| **失败隔离** | 单活动失败不影响其他，记录日志继续 |
| **并发控制** | 单线程串行执行，避免 LLM API 限流 |

## 7. Redis 缓存方案

### 7.1 缓存层次

```
请求 → Redis 缓存层 → MySQL 数据库
         │
         ├─ 命中 → 直接返回（< 5ms）
         └─ 未命中 → 查 DB → 写回缓存 → 返回
```

### 7.2 缓存 Key 设计

| 缓存 Key | 内容 | TTL | 说明 |
|----------|------|-----|------|
| `analytics:activity:{activityId}` | `AnalysisResponse` JSON | 24h | 单个活动分析 |
| `analytics:organizer:summary:{userId}` | `OrganizerAnalyticsSummary` JSON | 1h | 组织者总览 |
| `analytics:scheduled:lock` | 分布式锁标记 | 30min | 防止定时任务重复执行 |

### 7.4 注意事项

- 分析结果变更频率极低（仅定时任务写入），写入时更新缓存，平时只读
- TTL 设为 24h，保证定时任务跑完后缓存自动刷新
- 不使用 Redis 分布式锁（单实例部署场景），若扩展到多实例需引入 Redisson

### 8.2 超时与重试策略

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 连接超时 | 5s | TCP 握手超时 |
| 读取超时 | 30s | LLM 生成响应超时 |
| 最大重试次数 | 3 | HTTP 调用失败时 |
| 重试间隔 | 2s / 5s / 10s | 指数退避 |
| 降级策略 | 规则模板 | 3 次重试均失败 → fallback |

### 8.3 降级方案

当 LLM 调用失败时，`SuggestionGenerator.fallbackSummarize()` 根据指标阈值自动生成规则建议：

```java
private List<SuggestionItem> fallbackSummarize(ActivityMetrics metrics) {
    List<SuggestionItem> suggestions = new ArrayList<>();
    String seq = "1";

    if (metrics.signupRate() != null && metrics.signupRate().compareTo(BigDecimal.valueOf(50)) < 0) {
        suggestions.add(SuggestionItem.of("id-"+seq++, "promotion", "high",
            STR."活动报名率仅\{metrics.signupRate()}%，建议增强多渠道宣传推广。"));
    }
    if (metrics.attendanceRate() != null && metrics.attendanceRate().compareTo(S("70")) < 0) {
        suggestions.add(SuggestionItem.of("id-"+seq++, "schedule", "medium",
            STR."到场率仅\{metrics.attendanceRate()}%，建议提前推送签到提醒。"));
    }
    if (metrics.avgRating() != null && metrics.avgRating().compareTo(S("4.0")) < 0) {
        suggestions.add(SuggestionItem.of("id-"+seq++, "content", "high",
            STR."平均评分\{metrics.avgRating()}/5.0，建议收集低分反馈针对性优化内容。"));
    }
    if (suggestions.isEmpty()) {
        suggestions.add(SuggestionItem.of("id-1", "other", "low",
            "活动整体表现良好，可继续保持当前运营策略。"));
    }
    return suggestions;
}
```

## 11. 安全与隐私

| 措施 | 说明 |
|------|------|
| **API Key 安全** | `DEEPSEEK_API_KEY` 通过环境变量注入，不入库、不入版本控制 |
| **用户信息脱敏** | 传入 LLM 的文字评价已剥离用户 ID 和姓名，仅保留评分+内容 |
| **接口鉴权** | `/api/v1/analytics/**` 纳入 Spring Security 保护，组织者只能查看自己的活动分析 |
| **请求限流** | LLM API 调用通过定时任务串行执行，避免高频触发 |
| **日志审计** | `analysis_schedule_log` 记录每次任务执行状态、耗时和错误信息 |
