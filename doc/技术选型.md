### 3. 活动签到

- **需求说明**：系统支持活动参与者通过多种方式进行签到，包括二维码签到、定位签到和动态口令签到。活动组织者可实时查看签到情况并统计到场数据。

#### （1）二维码签到

- **技术选型**：ZXing 二维码生成库 + Redis 缓存

- **解决思路**：

  **二维码生成**：组织者进入签到页面时，前端自动向后端请求生成二维码。后端生成一个唯一标识 `checkin_token`（UUID）作为本次签到会话的凭证，构造包含活动 ID、token 和时间戳的二维码内容，使用 ZXing 库生成二维码图片返回前端展示。同时将 token 与活动 ID 关联存入 Redis，设置 30 秒有效期。

  **动态刷新**：前端每隔 30 秒自动请求刷新二维码，每次刷新生成新 token，旧 token 立即失效，有效防止截图盗用。

  **扫码验证**：参与者扫码后，前端解码得到活动 ID 和 token，发送至后端签到接口。后端验证 token 是否存在且未过期、用户是否已报名、是否已签到、是否在签到时间窗口内。全部通过后创建签到记录，标记签到成功，并删除 Redis 中的 token（防止重复使用）。

#### （2）定位签到

- **技术选型**：
  - **前端定位**：高德地图 JavaScript API 的 `AMap.Geolocation` 插件；
  - **前端地图选点**：高德地图 JavaScript API 的 `AMap.Map` 组件，结合 `react-amap` 或原生 API；
  - **距离计算**：后端使用 Haversine 公式计算球面距离（推荐），也可调用高德 Web 服务 API 进行距离验证；
  - **逆地理编码**：高德地图 Web 服务 API，用于将经纬度转换为结构化地址，便于签到记录留痕。

- **解决思路**：

  **活动地点配置**：组织者在创建或编辑活动时，通过高德地图组件选择地点，支持搜索地点或直接点击地图选点。选点后通过逆地理编码将经纬度转换为结构化地址，存入 MySQL 活动表，同时设置签到判定半径（默认 200 米）。

  **签到请求**：参与者点击定位签到按钮后，前端通过高德 `AMap.Geolocation` 插件获取当前精确位置（启用高精度定位），将经纬度与活动 ID 一并发送至后端。

  **距离验证**：后端从数据库获取活动经纬度和判定半径，采用 Haversine 公式计算用户当前位置与活动地点的球面距离，若距离在判定半径内则视为签到成功。

  **签到记录**：验证通过后创建签到记录，记录签到方式为 `location`，可选记录签到坐标供后续审计使用。

#### （3）动态口令签到

- **技术选型**：
  - **TOTP 算法**：RFC 6238 标准的一次性密码算法；
  - **Java 库**：jotp 或 Hutool 的 TOTP 工具类；
  - **缓存**：Redis（存储口令生成种子与状态）。

- **解决思路**：

  **口令生成**：组织者进入签到页面时，后端为该活动生成一个唯一的 Base32 编码密钥种子 `secret`，存入 Redis（有效期 30 秒）。基于 TOTP 算法，使用当前时间步长（默认 30 秒）和 HMAC-SHA1 签名生成 6 位动态口令，返回前端展示。

  **动态刷新**：前端每隔 30 秒自动请求刷新口令，基于相同的 `secret` 和当前时间步长重新计算，旧口令自动失效。

  **口令验证**：参与者输入口令后，前端将口令与活动 ID 发送至后端。后端从 Redis 获取 `secret`，基于当前时间步长重新计算期望口令，并允许前后各 1 个步长的容错（应对时钟偏差）。同时验证报名状态、签到状态和签到时间窗口，全部通过后创建签到记录，标记成功。

#### 签到统计

组织者可实时查看签到人数、未签人数、签到率及签到明细列表（含签到时间、签到方式），支持导出签到数据。


### 4. 单点登录集成

- **需求说明**：与上海交通大学 jAccount 单点登录系统（基于 OAuth 2.0 / OIDC 协议）集成。用户可通过 jAccount 账号一键登录，系统自动获取其学工号、姓名、身份类别等信息并建立本地会话。需与现有本地数据库登录方案共存。

- **技术选型**：Spring Security（已集成于 Spring Boot）

- **解决思路**：

  **认证流程**：
  1. **授权请求**：用户点击“jAccount 登录”，后端将请求重定向至 jAccount 授权服务器（携带 `client_id`、回调地址 `redirect_uri`、`scope=openid`）。
  2. **获取授权码**：用户在 jAccount 登录页完成认证并授权，授权服务器重定向至回调地址并附带一次性授权码 `code`（有效期 1 分钟）。
  3. **换取令牌**：后端携带 `code`、`client_id` 和 `client_secret` 向 jAccount 令牌端点发起请求，换取 `access_token`（有效期 30 分钟）、`refresh_token` 和 `id_token`。
  4. **解析用户信息**：使用 `client_secret` 验证 `id_token` 的签名和有效期，从 JWT 负载中提取 `sub`（jAccount 账号，唯一标识）、`name`（真实姓名）、`code`（学工号）和 `type`（身份类别）。
  5. **建立本地会话**：根据 `sub` 查询或创建本地用户记录，生成本地 JWT 返回前端完成登录。同时将 `access_token` 和 `refresh_token` 存入 Redis，关联当前用户，以便后续刷新或调用扩展接口获取更多信息。
  6. **登出**：清除本地会话，调用 jAccount 登出接口统一注销。

  **与本地登录的共存方案**：
  - 在 Spring Security 中配置多个 `AuthenticationProvider`，同时支持本地用户名/密码认证和 OAuth2 认证两种方式；
  - 用户可通过统一登录页面自由选择“本地账号登录”或“jAccount 登录”；
  - 两种方式共享同一套授权管理（`AuthorizationManager`）和权限控制组件；
  - 本地用户表增加 `jaccount` 字段存储用户的 jAccount 唯一标识，实现账号映射。

  > 借助 Spring Security 的 `spring-boot-starter-oauth2-client` 自动配置，可大幅简化集成工作，仅需在配置文件中声明 `provider` 和 `registration` 元数据。

- **注意事项**：
  - `client_id` 和 `client_secret` 需提前在“交我办”申请，并配置合法回调地址；
  - 敏感配置须通过环境变量注入，严禁硬编码；
  - 生产环境须配置 HTTPS，确保回调地址为 `https` 协议。


### 5. 智能分析与建议

- **需求说明**：对活动的报名转化率、到场率、用户评价（评分 + 文字反馈）进行智能分析，自动生成下一次活动的改进建议。分析维度可扩展至活动类别、时间、地点等。

  > 原需求中的“传播路径”因缺乏显式社交关系数据支撑，暂不纳入实现范围。若后续补充埋点数据（如分享链接追踪），可扩展该模块。

- **技术选型**：
  - **数据分析**：Spring Boot + MySQL（统计查询）；
  - **缓存**：Redis（缓存统计结果，减轻数据库压力）；
  - **LLM API**：DeepSeek API等，通过 HTTP 调用生成改进建议；
  - **定时任务**：Spring `@Scheduled` 或 Quartz（定期触发分析任务）。

- **解决思路**：

  **指标计算（离线/定时）** ：活动结束后，系统从数据库统计以下核心指标——报名转化率（报名人数 / 浏览量）、到场率（签到人数 / 报名人数）、平均评分（1-5 分）、评分分布（各分数段占比）。同时收集辅助维度：活动类别、时间段、地点、报名/签到的时间趋势、同类活动历史平均指标对比等。

  **数据摘要构建**：将统计结果格式化为结构化文本摘要，作为 LLM 的输入上下文，涵盖活动基本信息、各项指标数据及对比分析。

  **改进建议生成**：将数据摘要与预置提示词模板拼接，调用 LLM API 生成 3-5 条具体可行的改进建议，覆盖活动宣传、时间安排、场地设施、内容质量等维度。LLM 返回的 JSON 结构经解析后存入数据库，与对应活动关联。

  **结果展示**：组织者在活动详情页或分析看板中查看核心指标数值卡片、评分分布图表及 LLM 生成的改进建议列表。

- **注意事项**：
  - LLM 调用存在延迟和成本，建议采用**离线定时触发**（如活动结束后次日凌晨），避免同步调用影响用户体验；
  - 设置合理的超时与重试机制，LLM 调用失败时降级为规则模板生成；
  - API Key 须通过环境变量注入；
  - 用户文字评价涉及隐私，传入 LLM 前需做脱敏处理。


### 6. 社区聚类

- **需求说明**：根据师生的行为日志和个人信息进行聚类，自动划分用户社区并进行可视化展示。

  - **普通用户**：查看自己所属的社区、同社区其他用户、社区关联活动推荐及社区共享留言板；
  - **管理员**：查看所有社区的概览信息（名称、人数、特征标签）及详细成员列表；
  - **数据来源**：用户画像（兴趣标签、学院年级）+ 行为日志（报名、签到、评价、浏览记录）。

- **技术选型**：
  - **聚类算法**：K-Means（sklearn 或 Java 实现）；
  - **Python 服务**：Flask / FastAPI（提供聚类计算接口，供 Spring Boot 调用），或直接调用 Python 脚本；
  - **特征工程**：Python（pandas、numpy）；
  - **可视化**：ECharts（散点图 / 关系图）+ React。

- **解决思路**：

  **特征工程（离线）** ：将每个用户转化为数值型特征向量，维度包括：兴趣标签（多热编码）、学院年级（One-Hot 编码）、参与频次（归一化数值）、活跃时段（One-Hot 编码）、社交共现（降维向量）。最终得到 `N × d` 的特征矩阵（`N` 为用户总数）。若维度过高，可先通过 PCA 降维至 50-100 维。

  **K-Means 聚类（离线）** ：将 N 个用户划分为 K 个社区（K 默认 5-8，管理员可配置）。算法迭代分配用户至最近的聚类中心并更新中心位置，直至收敛。聚类数可通过肘部法则辅助确定。

  **社区更新策略**：聚类结果会因新用户加入和行为变化而动态更新，用户被重新划分时需保障体验：
  - **版本化归属**：每次聚类生成新版本号，保留历史归属记录供追溯；
  - **归属冻结期**：对已有稳定归属的用户设置过渡期，期间仍显示旧社区，通过系统通知提前告知；
  - **平滑迁移**：迁移后在新社区留言板自动发布欢迎通知，推荐流逐步过渡至新社区特征，旧社区内容保留只读权限；
  - **频率控制**：聚类任务每周/每两周执行一次，安排在凌晨低峰期。

  **可视化展示**：
  - 使用 t-SNE 或 PCA 将高维向量降维至 2D 平面，通过 ECharts 散点图展示用户分布，不同颜色代表不同社区；
  - 社区详情页展示社区名称（自动生成）、人数、核心特征标签、成员列表、关联活动推荐及共享留言板；
  - 管理员看板展示所有社区概览统计及社区间相似度矩阵。

  **共享留言板**：每个社区拥有独立留言板，仅限本社区成员查看和发帖，支持回复功能。用户迁移后，旧社区留言保留只读权限，新社区留言板授予发帖权限。


### 7. 热度计算与更新（隐性需求）

- **需求说明**：为每个活动设计动态热度值，综合反映活动的受欢迎程度与时效性，并按预设周期定时更新。热度值用于活动列表排序、推荐召回兜底、首页热门展示等场景。

- **技术选型**：
  - **定时任务**：Spring Boot `@Scheduled`（或 Quartz）；
  - **存储**：MySQL（持久化热度值）+ Redis（存储热度排行榜 `ZSET`，支持高效排序查询）；
  - **计算引擎**：Java 原生实现，无需额外依赖。

- **解决思路**：

  **热度因子设计**：热度由四个维度加权计算——浏览量（权重 0.15）、报名数（权重 0.35）、签到数（权重 0.30）、收藏数（权重 0.20）。各因子权重可根据实际运营情况调整。

  **时间衰减机制**：采用类 Hacker News 衰减算法，以活动发布时间为基准，新活动热度更高。发布后 0-12 小时衰减缓慢，12-72 小时逐步下降，72 小时后趋于平稳但仍有剩余热度。

  **活动状态加成**：根据活动当前状态进行修正——未开始且距开始 > 48 小时（× 0.8），未开始且距开始 ≤ 48 小时（× 1.3），进行中（× 1.5），已结束 ≤ 7 天（× 0.7），已结束 > 7 天（× 0.3），避免已结束活动长期霸榜。

  **定时更新策略**：每小时执行一次全量更新。为提高效率，仅对发布 7 天以内、过去 1 小时有新动态、或状态为“进行中”的活动进行重算。更新后批量写入 MySQL，并同步更新 Redis 的 `hot:rank` 有序集合。

  **榜单应用**：首页热门推荐直接从 Redis 的 `ZSET` 按分数倒序取 Top-N 活动；用户可按热度排序，接口通过 `REVRANGE` 分页获取活动 ID 后联查 MySQL 补全详情；推荐冷启动时以热度最高的活动作为默认召回结果。
