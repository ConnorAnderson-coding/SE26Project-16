# 语义检索与智能推荐核心算法流程

**活动发现与个性化推送 — 功能设计与算法流程**

---

## 一、功能定位

为校园活动平台提供两大核心能力，共享 Elasticsearch 8.x 集群、GTE 稠密向量模型（`campus_gte`）与活动索引 `campus_activities`：

| 能力 | 用户场景 | 核心问题 |
|------|----------|----------|
| **语义检索** | 活动列表页输入自然语言或关键词 | 从海量活动中找出**与 query 相关**的结果 |
| **智能推荐** | 首页「为你推荐」 | 根据兴趣、行为、社交、时间等找出**用户可能想参加**的活动 |

**存储分工**：

- **MySQL**：唯一数据源（Source of Truth），存活动、用户、报名等全量业务数据。
- **Elasticsearch**：语义检索与推荐召回共用的派生索引 / 向量。
- **Redis**：活动详情 / 热门列表缓存；用户喜好向量缓存。

检索实现在 `search` 包，推荐实现在 `recommend` 包。热度分（`hotness_score`）由 `ActivityHotnessService` 独立维护，详见 [`热度算法流程.md`](热度算法流程.md)。

---

## 二、整体设计思路

四个核心原则：

1. **MySQL 主库 + ES 派生索引** — 活动 CRUD 以 MySQL 为准；创建/编辑活动时同步 ES 并经由 ingest pipeline 向量化，检索与推荐均从 ES 召回、MySQL 回填详情。
2. **双路召回 + 统一打分** — 检索并行 BM25 关键词路与 GTE kNN 语义路，合并后做相关度打分；推荐以用户喜好向量 kNN 召回，再叠加多因子加权。
3. **硬约束前置、软特征后置** — 类别/状态/地点等过滤在 ES 层完成；已报名、时间冲突在推荐层硬过滤；热度、社交、标签等作为排序辅助特征。
4. **优雅降级** — ES 不可用、向量模型缺失、kNN 失败时，分别降级为 MySQL LIKE、纯 BM25、热门活动冷启动或规则推荐，保证功能可用。

## 三、时序流程

### 3.1 索引维护（离线 / 增量）

```
┌─────────────────────────────────────────────────────────────┐
│ 活动创建 / 编辑 / 删除（ActivityService）                    │
│   └─ syncSearchIndex → ActivityIndexService                 │
│       ├─ draft 活动 → 从 ES 删除                            │
│       └─ 已发布活动 → index + pipeline campus-activity-embedding │
│           search_text → GTE → activity_embedding (512-d)    │
├─────────────────────────────────────────────────────────────┤
│ 应用启动（ActivitySearchIndexBootstrap）                     │
│   └─ 索引为空且 auto-rebuild-on-startup=true → 全量重建     │
├─────────────────────────────────────────────────────────────┤
│ 管理员手动触发                                               │
│   └─ POST /api/v1/search/index/rebuild → bulk 全量索引      │
└─────────────────────────────────────────────────────────────┘
```

### 3.2 语义检索（在线）

```
┌─────────────────────────────────────────────────────────────┐
│ 用户请求 keyword + 过滤条件 + sort                           │
├─────────────────────────────────────────────────────────────┤
│ ES 未启用 / 索引为空 → MySQL LIKE 降级                       │
├─────────────────────────────────────────────────────────────┤
│ ActivitySearchService                                        │
│   ├─ 路 A：BM25 multi_match → Top-N + keywordScore         │
│   ├─ 路 B：GTE kNN on activity_embedding → Top-N + semanticScore │
│   ├─ SemanticScoreFilter：仅语义命中需 semanticScore ≥ τ   │
│   ├─ HybridRelevanceScorer：合并两路 → relevance（searchScore）│
│   ├─ hydrate：ES 召回 ID → MySQL 批量回填 ActivityResponse  │
│   └─ ActivitySearchRanker：按用户 sort 二次排序 + 内存分页    │
└─────────────────────────────────────────────────────────────┘
```

### 3.3 智能推荐（在线）

```
┌─────────────────────────────────────────────────────────────┐
│ GET /api/v1/activities/recommended?limit=M                   │
├─────────────────────────────────────────────────────────────┤
│ 1. 加载用户资料 + 报名记录（排除 rejected）                  │
│ 2. UserPreferenceVectorService → 用户喜好向量 v_u            │
│    失败 → 冷启动（无 kNN）                                   │
│ 3. ES kNN：v_u → Top-N 候选 + sim 分                        │
│    失败 → 冷启动：MySQL 热度 Top-N                           │
│ 4. RecommendationScorer.hardFilter                           │
│    剔除已报名 + 与已报名活动时间冲突的候选                    │
│ 5. 候选不足 (<12) → 扩大 kNN → 兴趣标签回填                  │
│ 6. SocialAffinityService → 用户与各组织者的社交强度           │
│ 7. RecommendationScorer.scoreAndRank → finalScore          │
│ 8. 取 Top-M，输出 recommendScore (0–100) + recommendReasons │
├─────────────────────────────────────────────────────────────┤
│ ES / 推荐链路异常 → ActivityService.getRecommendedLegacy     │
│   热度 Top-50 + 标签匹配规则分                               │
└─────────────────────────────────────────────────────────────┘
```

## 四、索引构建算法

**目标**：将 MySQL 活动数据转化为 ES 可检索、可 kNN 的文档。

**输入**：`Activity` 实体

**输出**：ES 文档（含 `activity_embedding` 512 维向量）

### 4.1 文档字段

| 字段 | 类型 | 用途 |
|------|------|------|
| `title`, `description`, `search_text` | text + IK | BM25 关键词检索 |
| `tags`, `category`, `location`, `status` | keyword | 过滤 / 精确匹配 |
| `activity_embedding` | dense_vector 512 / cosine | GTE 稠密向量（kNN） |
| `signup_count`, `favorite_count` | integer | 索引侧冗余（排序以 MySQL `hotness_score` 为准） |
| `start_time`, `end_time` | date | 时间信息 |

### 4.2 search_text 拼接规则

由 `ActivityDocumentMapper` 拼接：

```
search_text = title + description + category（含中文类别名）+ tags + location
```

**刻意排除** `college`（避免「计算机学院」类归属信息导致「计算机科学」误召回无关活动）。`college` 仍作为独立 keyword 字段可过滤。

### 4.3 向量化流程

1. 后端写入 ES 文档时挂载 ingest pipeline `campus-activity-embedding`。
2. Pipeline 调用 GTE 模型（`thenlper/gte-small-zh`，512 维）对 `search_text` 推理。
3. 结果写入 `activity_embedding`（**无** E5 式 `passage:` 前缀）。

**在线 query 向量化**：kNN 使用 `query_vector_builder.text_embedding`，`model_text` 为原始 keyword（**无** `query:` 前缀）。

### 4.4 可索引条件

- `status != draft` 的活动写入索引。
- 草稿或已删除活动从 ES 移除。

## 五、语义检索算法

**目标**：根据用户 query 从 ES 召回相关活动，结合 MySQL 热度等字段按用户指定方式排序。

**输入**：`keyword`、可选 `category/status/location` 过滤、`mode`、`sort`、`matchWeight`

**输出**：分页 `ActivityResponse` 列表（含 `searchScore`、`keywordScore`、`semanticScore`）

### 5.1 检索模式

| mode | 行为 |
|------|------|
| `keyword` | 仅 BM25 |
| `semantic` | 仅 kNN；失败或空结果降级 BM25 |
| `hybrid`（默认） | BM25 + kNN 并行 → 筛除 → 加权融合 |

每路召回 Top-N，默认 N = 50（`search-recall-size`）。

### 5.2 BM25 关键词检索

**原理**：基于词频 TF、逆文档频率 IDF 与文档长度归一化的经典排序。

**ES 查询**（`ElasticsearchActivitySearchRepository`）：

```
multi_match:
  fields: title^3, description^2, search_text, tags
  analyzer: IK 分词（索引 ik_max，检索 ik_smart）
filter: category / status / location；排除 draft
```

**特点**：精确字面匹配；「羽毛球」能命中标题含该词的活动；「周末放松」若无字面匹配则 0 条。

### 5.3 GTE 稠密向量语义检索

**原理**：`campus_gte` 将中文文本编为 512 维向量，用 cosine 相似度做 kNN。

**ES kNN 参数**：

```
field: activity_embedding
k = recallSize
numCandidates = max(recallSize × 2, 100)
model_id = campus_gte
model_text = keyword
filter: status=published + 可选 category/status/location
```

**分数映射**：ES cosine 下 `_score` 一般为 `(1 + cosine) / 2`，落在约 `[0, 1]`，即 API 的 `semanticScore`。

**特点**：「周末放松」可召回无共同关键词但语义相近的活动。

### 5.4 语义绝对阈值筛除

**问题**：稠密向量分数整体偏高，弱相关向量命中不应进入最终结果。

**策略**（`SemanticScoreFilter`，默认 τ = 0.90）：

| 候选来源 | 处理 |
|---------|------|
| BM25 已命中 | **始终保留**（豁免阈值） |
| 仅语义召回 | 保留当且仅当 `semanticScore ≥ τ` |

配置：`app.elasticsearch.semantic-absolute-threshold=0.90`（约合 cosine ≥ 0.80）。

### 5.5 加权相关度融合（Hybrid）

**问题**：BM25 与语义分量纲不同，需统一为可排序的相关度分。

**策略**（`HybridRelevanceScorer`）：

```
bm25Norm = bm25(d) / max_bm25(本轮召回)     # 无 BM25 命中则为 0
sem      = semanticScore(d) 或 0

若 BM25 命中:
  relevance = λ_bm25 × bm25Norm + λ_sem × sem + λ_bonus
否则（仅语义，且已过阈值）:
  relevance = λ_sem_only × sem
```

默认权重：

| 参数 | 配置项 | 默认 |
|------|--------|------|
| λ_bm25 | `relevance-bm25-weight` | 0.50 |
| λ_sem | `relevance-semantic-weight` | 0.35 |
| λ_bonus | `relevance-keyword-bonus` | 0.15 |
| λ_sem_only | `relevance-semantic-only-weight` | 0.90 |

**效果**：关键词命中档整体高于纯语义档；同档内由 BM25 与语义微调顺序。`searchScore` = `relevance`。

### 5.6 MySQL 回填与二次排序

1. 按 ES 召回顺序取 `activityId`。
2. `activityRepository.findAllById` 批量加载 MySQL 活动详情。
3. 写入 `searchScore`、`keywordScore`、`semanticScore`、`searchChannel`。
4. ES 有但 MySQL 缺失的文档跳过。

**后端排序**（`ActivitySearchRanker`，在召回全集上排序后内存分页）：

| sort | 行为 |
|------|------|
| `relevance`（默认） | 按 `searchScore` 降序 |
| `hot` | 按 `hotnessScore` 降序 |
| `time` | 按 `startTime` 升序 |
| `signup` | 按 `signupCount` 降序 |
| `composite` | `w × norm(相关度) + (1-w) × norm(热度)`，默认 w=0.7 |

归一化：在当前召回列表内 min-max；若 max=min 则 norm=1.0。前端「综合匹配」Slider 调节 `matchWeight`（0–1）。

## 六、用户喜好向量算法

**目标**：将用户兴趣标签与近期参与行为融合为 512 维喜好向量，供推荐 kNN 召回。

**输入**：用户 `interests`、最近 K 条报名记录

**输出**：L2 归一化后的 `float[512]`，或 `null`（触发冷启动）

### 6.1 数据采集来源

| 来源 | 内容 |
|------|------|
| MySQL `user.interests` | 兴趣标签 JSON 数组 |
| MySQL `registration` | 最近 K=10 条非 rejected 报名 |
| ES `activity_embedding` | 各报名活动对应的向量（mget） |
| Redis `campus:user:pref_vector:{userId}` | 缓存（TTL 6h） |

### 6.2 兴趣侧向量（vec₁）

```
text = join(interests, " ")
vec₁ = normalize(GTE_infer(text))
```

兴趣为空或 infer 失败时，`vec₁ = null`。

### 6.3 历史行为侧向量（vec₂）

对最近 K 条报名活动：

```
w_i = 0.5^(daysAgo_i / halfLife)     # 默认 halfLife = 30 天
vec₂ = normalize(Σ w_i × embedding_i)
```

缺向量的活动跳过；全部缺失时 `vec₂ = null`。

### 6.4 凸组合

| 条件 | 结果 |
|------|------|
| vec₁、vec₂ 均有 | `v_u = normalize(α × vec₁ + (1-α) × vec₂)`，默认 α=0.4 |
| 仅 vec₁ | `v_u = vec₁` |
| 仅 vec₂ | `v_u = vec₂` |
| 均为 null | 返回 null → **冷启动** |

### 6.5 缓存失效

以下操作调用 `UserPreferenceVectorService.invalidate(userId)`：

- 用户修改兴趣标签
- 报名成功
- 报名审核结果变更

## 七、智能推荐算法

**目标**：在硬过滤后的候选集上，综合内容相似度与多因子辅助特征，输出 Top-M 推荐结果。

**输入**：用户 ID、limit M

**输出**：`ActivityResponse` 列表（含 `recommendScore`、`recommendReasons`）

### 7.1 召回

**正常路径**：ES kNN，`query_vector = v_u`，filter `status=published`，默认 N=50。

**冷启动**（无喜好向量或 kNN 失败）：MySQL 按 `hotness_score` 降序取 Top-N，sim=0。

### 7.2 硬过滤

`RecommendationScorer.hardFilter` 剔除：

1. **已报名**的活动（`signedUpIds`）。
2. **时间冲突**的候选：`start < otherEnd ∧ end > otherStart`（与任一已报名活动时间窗口重叠）。

仅保留 `status=published` 的候选。

### 7.3 补召回

过滤后候选数 < 12（`recommend-min-candidates`）时：

1. 将 kNN 扩大到 N=100 重试。
2. 仍不足：从热度 Top 活动中回填**兴趣标签命中**的已发布活动，sim 先验 = `min(当前 sim) - 0.02`（默认 0.85）。

### 7.4 社交强度

对用户 A 与活动组织者 B（B ≠ A）：

```
s(A,B) = log(1 + c_co) + 0.8×log(1 + c_A→B) + 0.8×log(1 + c_B→A)
```

| 符号 | 含义 |
|------|------|
| c_co | 共同报名活动数（非 rejected） |
| c_A→B | A 报名 B 组织的活动数 |
| c_B→A | B 报名 A 组织的活动数 |

由 MySQL 实时聚合，语料规模小，无需预计算共现表。

### 7.5 最终推荐分（内容优先）

针对「窄 sim 分布 + 全零 tag 被 min-max 刷成满分」问题，采用内容优先打分：

**非冷启动**：

```
final = simComponent + ε × auxComponent

ε = 0.25（recommend-aux-scale）

auxComponent = (wTag×tagN + wSoc×socN + wHot×hotN + wTime×time) / wAux
```

| 因子 | 含义与归一化 |
|------|-------------|
| simComponent | 候选 sim 跨度 ≤ δ=0.03 时用**绝对** ES 分；跨度更大时做候选内 min-max |
| tagN | min(1, tagHits / tagCap)，tagCap=2；**不对全零做 min-max** |
| socN | min(1, social / socialRef)，socialRef=3.5 |
| hotN | 候选内 min-max(`hotnessScore`)；无 spread 时输出 0 |
| time | 是否落在 `available_time`（0/1；未设置则 1） |

默认权重：wTag=0.15，wSoc=0.05，wHot=0.10，wTime=0.05。

**冷启动**：

```
wHot += wSim（0.10 + 0.55 = 0.65）
final = auxComponent（simComponent 全 0）
```

### 7.6 展示分与推荐理由

```
recommendScore = round(clamp01(final / denom) × 100)

非冷启动: denom = 1 + ε = 1.25
冷启动:   denom = 1.0
```

**推荐理由**（最多 3 个中文标签，按贡献度排序）：

| 标签 | 触发条件 |
|------|---------|
| 兴趣匹配 | tagN ≥ 0.5 |
| 内容相似 | sim 高于中位数或 simComponent ≥ 0.92 |
| 社交相关 | socN ≥ 0.25 |
| 热门活动 | hotN ≥ 0.55 |
| 时间合适 | 用户设置了 available_time 且 time=1 |
| 热门推荐 | 冷启动兜底 |

## 八、降级与边界条件

| 场景 | 处理策略 |
|------|---------|
| `ES_ENABLED=false` | 搜索/推荐/索引 Service 不注册；列表带 keyword 走 MySQL LIKE |
| ES 索引为空 | 警告 + MySQL LIKE 降级 |
| 语义通路失败（SEMANTIC） | 降级 BM25 |
| 语义通路失败（HYBRID） | 仅 BM25 |
| GTE 模型未部署 | kNN / infer 抛 BusinessException 或返回 null |
| 喜好向量构建失败 | 冷启动（热度 Top-N） |
| 推荐 kNN 失败 | 冷启动 |
| 硬过滤后候选为空 | 返回空列表 |
| `ActivityService.getRecommended` 异常 | `getRecommendedLegacy`：热度 Top-50 + `tagMatch×30 + min(hotness×20, 80)` |
| ES 启动 bootstrap 失败 | 最多重试 6 次，需手动 rebuild |

### 9.1 主要 API

| 方法 | 路径 | 说明 |

### 9.2 响应字段（检索相关）

| 字段 | 含义 |
|------|------|

| `recommendScore` | 推荐展示分（0–100） |

| `app.elasticsearch.semantic-absolute-threshold` | 0.90 | 语义-only 阈值 τ |
| `app.elasticsearch.relevance-bm25-weight` | 0.50 | hybrid BM25 权重 |
| `app.elasticsearch.relevance-semantic-weight` | 0.35 | hybrid 语义权重 |
| `app.elasticsearch.relevance-keyword-bonus` | 0.15 | BM25 命中 bonus |
| `app.elasticsearch.relevance-semantic-only-weight` | 0.90 | 纯语义缩放系数 |
| `app.elasticsearch.pref-history-size` | 10 | 偏好历史条数 K |
| `app.elasticsearch.pref-half-life-days` | 30 | 报名衰减半衰期 |
| `app.elasticsearch.pref-interest-mix` | 0.4 | 兴趣/历史凸组合 α |
| `app.elasticsearch.recommend-recall-size` | 50 | 推荐 kNN k |
| `app.elasticsearch.recommend-min-candidates` | 12 | 补召回阈值 |
| `app.elasticsearch.recommend-aux-scale` | 0.25 | 辅助分缩放 ε |
| `app.elasticsearch.recommend-weight-hot` | 0.10 | 热度权重 |

## 十、三库协作小结

| 存储 | 检索模块 | 推荐模块 |
|------|----------|----------|
| **MySQL** | 活动主数据；ID 回填；`hotnessScore` 排序 | 报名/冲突过滤、社交聚合、冷启动热门、详情回填 |
| **Elasticsearch** | BM25 + GTE kNN + τ + 加权相关度 | 活动 embedding mget / 兴趣 infer / 喜好向量 kNN |
| **Redis** | 热门列表、详情缓存 | 用户喜好向量缓存（6h TTL） |

## 十二、验证方法

### 12.1 快速功能验证

| 查询 | mode | 预期 |
|------|------|------|
| 羽毛球 | keyword | 字面匹配，约 1 条 |
| 周末放松 | keyword | 0 条（无字面匹配） |
| 周末放松 | semantic / hybrid | 语义相关结果；仅语义命中需 `semanticScore ≥ 0.90` |
| 羽毛球 | hybrid + sort=relevance | 羽毛球赛排第一 |
| 周末放松 | hybrid + sort=hot | 顺序按 `hotnessScore`，与 relevance 不同 |
| 羽毛球 | hybrid + sort=composite + matchWeight=0 | 等同纯热度排序 |
