# 活动热度计算核心算法流程

**动态排序与推荐辅助 — 功能设计与算法流程**

---

## 一、功能定位

为校园活动平台提供统一的**活动综合热度分**（`hotness_score`），综合反映活动的受欢迎程度与时效性。热度值用于：

- 活动列表按热度排序（MySQL / ES 检索结果二次排序）
- 首页热门活动展示
- 智能推荐系统的辅助特征与冷启动兜底
- ES 不可用时的规则推荐降级

热度计算由 `ActivityHotnessService` 统一负责，结果持久化至 MySQL，并同步写入 Redis 有序集合，供快速排行榜查询。

---

## 二、整体设计思路

四个核心原则：

1. **单一数据源** — 所有下游场景（检索、推荐、首页）均读取同一份 `hotness_score`，避免多处各自计算导致排序不一致。
2. **分层计算** — 热度 = 基础互动分 × 时间衰减 × 状态修正，三层依次叠加，职责清晰、便于调参。
3. **事件 + 定时双驱动** — 用户互动（报名、收藏等）触发即时重算；每小时定时全量刷新，补偿时间衰减与状态变化。
4. **双写持久化** — 每次重算同时更新 MySQL（持久化）与 Redis ZSET（排行榜），读写路径分离。

## 三、时序流程

```
┌─────────────────────────────────────────────────────────────┐
│ 应用启动（ApplicationRunner）                                │
│   └─ calculateAndSaveHotnessScores()                        │
│       遍历全部活动 → 计算热度 → 写 MySQL + Redis ZSET        │
├─────────────────────────────────────────────────────────────┤
│ 每小时整点（@Scheduled cron = "0 0 * * * ?"）               │
│   └─ calculateAndSaveHotnessScores()                        │
│       全量重算所有活动（含时间衰减、状态修正的被动变化）       │
├─────────────────────────────────────────────────────────────┤
│ 用户互动事件（即时触发 recalculate）                         │
│   ├─ 创建活动 → ActivityService.create                      │
│   ├─ 用户报名 → RegistrationService.signup                  │
│   ├─ 收藏/取消收藏 → FavoriteService.toggle                 │
│   └─ 发布活动记录（标记 ended）→ ActivityRecordService.publish│
├─────────────────────────────────────────────────────────────┤
│ 下游消费                                                     │
│   ├─ 检索排序：ActivitySearchRanker（hot / composite）      │
│   ├─ 智能推荐：RecommendationScorer（hot 辅助分）           │
│   ├─ 冷启动兜底：RecommendationService.coldStartCandidates  │
│   └─ 规则推荐降级：ActivityService.getRecommendedLegacy     │
└─────────────────────────────────────────────────────────────┘
```

**说明**：当前定时任务为**全量重算**（`activityRepository.findAll()`），而非增量筛选。时间衰减与状态修正随时间被动变化，需依赖定时任务刷新；用户互动则通过事件触发即时更新。

## 四、热度计算算法

**目标**：将活动的原始互动数据转化为一个可比较、可排序的标量分数。

**输入**：`Activity` 实体（含互动计数、状态、时间字段）

**输出**：`hotness_score`（`double`，≥ 0）

### 4.1 数据采集来源

| 数据来源 | 字段 | 说明 |
|---------|------|------|
| 活动表 `activity` | `view_count` | 浏览量 |
| 活动表 `activity` | `signup_count` | 报名人数 |
| 活动表 `activity` | `check_in_count` | 签到人数 |
| 活动表 `activity` | `favorite_count` | 收藏人数 |
| 活动表 `activity` | `status` | 活动状态 |
| 活动表 `activity` | `start_time` / `end_time` | 活动时间区间 |
| 活动表 `activity` | `created_at` | 发布时间（衰减基准） |

### 4.2 计算公式（三阶段流水线）

```
hotness_score = applyStatusMultiplier(
                  applyTimeDecay(
                    calculateBaseHotness(activity),
                    created_at
                  ),
                  activity
                )
```

**前置短路**：

| 条件 | 返回值 |
|------|--------|
| `activity == null` 或 `created_at == null` | `0.0` |
| `status == "cancelled"` | `0.0` |

---

#### 阶段一：基础互动分（Base Hotness）

对四项互动指标做**线性加权求和**：

```
base = view_count   × 0.15
     + signup_count × 0.35
     + check_in_count × 0.30
     + favorite_count × 0.20
```

| 因子 | 权重 | 设计意图 |
|------|------|---------|
| 浏览量 | 0.15 | 反映曝光，权重最低（浅层互动） |
| 报名数 | 0.35 | 反映转化意愿，权重最高 |
| 签到数 | 0.30 | 反映真实参与，权重次高 |
| 收藏数 | 0.20 | 反映兴趣留存 |

空值处理：各计数字段为 `null` 时按 `0` 计。

---

#### 阶段二：时间衰减（Time Decay）

以活动**发布时间** `created_at` 为基准，采用**分段指数衰减**（类 Hacker News 思路），使新发布活动获得更高曝光优先级。

设 `h = hoursSinceCreation = now - created_at`（小时，负值截断为 0）：

| 时间区间 | 衰减乘数 |
|---------|---------|
| `h ≤ 12` | `0.99^h` |
| `12 < h ≤ 72` | `0.99^12 × 0.95^(h - 12)` |
| `h > 72` | `0.99^12 × 0.95^60 × 0.90^(h - 72)` |

```
decayed = base × decayMultiplier(h)
```

**衰减曲线特征**：

| 发布后时长 | 近似衰减乘数 | 含义 |
|-----------|-------------|------|
| 0 小时 | 1.000 | 无衰减 |
| 12 小时 | ≈ 0.886 | 缓慢下降 |
| 72 小时 | ≈ 0.041 | 显著下降 |
| 168 小时（7 天） | ≈ 0.003 | 趋于低位 |

72 小时后仍以 `0.90/小时` 继续衰减，但保留残余热度，不会归零。

---

#### 阶段三：状态修正（Status Multiplier）

根据活动当前生命周期状态，对衰减后的分数施加**乘性修正**，避免已结束活动长期占据热门榜单，同时放大临近/进行中的活动。

| 状态条件 | 乘数 | 常量名 |
|---------|------|--------|
| 未开始，距开始 > 48 小时 | × 0.8 | `MULTIPLIER_NOT_STARTED_GT_48H` |
| 未开始，距开始 ≤ 48 小时 | × 1.3 | `MULTIPLIER_NOT_STARTED_LE_48H` |
| 进行中 | × 1.5 | `MULTIPLIER_IN_PROGRESS` |
| 已结束 ≤ 7 天 | × 0.7 | `MULTIPLIER_ENDED_LE_7D` |
| 已结束 > 7 天 | × 0.3 | `MULTIPLIER_ENDED_GT_7D` |
| 已取消 | 直接返回 0 | — |
| 草稿 / 未知状态 | × 1.0（不修正） | — |

**状态判定逻辑**（`applyStatusMultiplier`）：

```
switch status:
  "published" / "upcoming":
    if start_time 存在:
      hoursUntilStart = start_time - now
      if hoursUntilStart > 48  → × 0.8
      elif hoursUntilStart > 0 → × 1.3
      elif now < end_time       → × 1.5（时间窗口内视为进行中）
  "in_progress" / "ongoing"  → × 1.5
  "ended":
    daysSinceEnd = now - end_time（天）
    if daysSinceEnd ≤ 7 → × 0.7
    else                → × 0.3
  default（draft 等）         → × 1.0
```

---

### 4.3 计算示例

假设某活动：浏览 100、报名 20、签到 10、收藏 5；发布 24 小时前；状态为 `published`，距开始 30 小时。

```
base     = 100×0.15 + 20×0.35 + 10×0.30 + 5×0.20
         = 15 + 7 + 3 + 1 = 26.0

decay    = 0.99^12 × 0.95^12 ≈ 0.886 × 0.540 ≈ 0.479
decayed  = 26.0 × 0.479 ≈ 12.45

multiplier = 1.3（未开始，距开始 ≤ 48h）
hotness_score ≈ 12.45 × 1.3 ≈ 16.19
```

## 五、持久化与缓存算法

**目标**：保证热度分写入可靠、读取高效。

### 5.1 写入流程（persistHotness）

```
1. hotnessScore = calculateHotness(activity)
2. activity.hotnessScore = hotnessScore
3. activityRepository.save(activity)          → MySQL 持久化
4. Redis ZSET.add("activity:hotness", id, score) → 排行榜同步
```

### 5.2 存储结构

| 存储 | Key / 字段 | 用途 |
|------|-----------|------|
| MySQL `activity.hotness_score` | 列值 | 持久化、JPA 查询排序、索引 `idx_activity_hotness` |
| Redis ZSET `activity:hotness` | member = 活动 ID，score = 热度分 | Top-N 排行榜、分页 REVRANGE |
| Spring Cache `activity:list:hot` | 分页缓存（TTL 10 分钟） | 热门列表接口加速 |

### 5.3 缓存失效策略

以下写操作会 `@CacheEvict(activity:list:hot, allEntries=true)` 清空热门列表缓存：

- 创建活动
- 更新 / 删除活动
- 报名
- 收藏切换
- 发布活动记录

## 六、下游消费算法

热度分计算完成后，被检索、推荐、首页等多个模块以不同方式消费。

### 6.1 检索排序（ActivitySearchRanker）

| 排序模式 | 算法 |
|---------|------|
| `hot` | 按 `hotnessScore` 降序 |
| `composite` | `score = w × norm(相关度) + (1-w) × norm(热度)`，默认 w=0.7 |
| `relevance` / MySQL 降级 | 默认按 `hotnessScore` 降序（无 ES 相关度时的兜底） |

其中 `norm(x) = (x - min) / (max - min)`，当 `max ≤ min` 时返回 `1.0`。

### 6.2 智能推荐（RecommendationScorer）

在 kNN 召回 + 硬过滤之后，热度作为辅助特征参与最终排序：

```
finalScore = simComponent + ε × auxComponent

auxComponent 中热度贡献：
  hotN[i] = min-max 归一化(hotnessScore)
  cHot    = (wHot × hotN[i]) / wAux
```

默认权重（`application.properties`）：`recommend-weight-hot = 0.10`。

**冷启动**：无用户行为向量时，sim 权重并入 hot（`wHot += wSim`），直接按热度 Top-N 召回。

**推荐理由**：归一化热度 ≥ 0.55 时标注「热门活动」。

### 6.3 规则推荐降级（getRecommendedLegacy）

ES / 智能推荐不可用时：

```
1. 取热度 Top 50（findPublishedByHot）
2. 过滤已报名活动
3. 叠加标签匹配分：tagMatch × 30 + min(hotness × 20, 80)
4. 按 recommendScore 降序返回
```

| 用户行为 | 更新字段 | 是否触发重算 |
|---------|---------|-------------|

## 八、设计权衡与边界条件

| 场景 | 处理策略 |
|------|---------|
| 活动已取消 | 热度强制为 0，不参与排行 |
| 草稿活动 | 不做状态修正（× 1.0），但通常不对外展示 |
| 计数为 null | 按 0 处理，不抛异常 |
| `created_at` 缺失 | 返回 0.0 |
| 定时任务失败 | 下次整点重试；事件触发的即时重算不受影响 |
| Redis 不可用 | MySQL 仍持久化；ZSET 写入失败需依赖运维恢复 |
| 全量 vs 增量重算 | 当前为全量，活动量大时可改为增量筛选优化 |
| 权重调参 | 常量硬编码于 `ActivityHotnessService`，未来可配置化 |

## 十、验证方法

1. **启动应用**：观察日志 `初始化活动综合热度分...`，确认 MySQL `hotness_score` 非空。
2. **模拟互动**：报名或收藏某活动，检查该活动 `hotness_score` 即时上升。
3. **Redis 校验**：`ZREVRANGE activity:hotness 0 9 WITHSCORES` 查看 Top 10 排行。
4. **检索验证**：搜索接口 `sort=hot` 与 `sort=composite`，确认顺序与热度分一致。
5. **时间衰减**：等待或修改 `created_at` 测试数据，触发定时任务后观察分数下降。
